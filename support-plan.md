# Support Plan

## Project: Python Task Manager App

---

## 1. Issue Reporting Process

Users or team members who encounter issues should report them using the following process:

### Step 1 — Identify the Issue
- Is it a bug (something broken)?
- Is it a feature request (something missing)?
- Is it a performance problem (something slow)?

### Step 2 — Report the Issue
- **Internal Team:** Create an issue in the GitHub repository with:
  - Clear title describing the problem
  - Steps to reproduce
  - Expected vs actual behavior
  - Screenshots if applicable
  - Severity: Critical / High / Medium / Low

- **End Users (if applicable):** Submit via email or support form with description and screenshot.

### Step 3 — Issue Triage
- The QA Engineer (Member D) reviews new issues within 1 business day
- Issue is assigned a severity level and an owner
- Added to the defect log if confirmed as a bug

---

## 2. Severity Levels & Response Times

| Severity | Description                              | Response Time | Resolution Time |
|----------|------------------------------------------|---------------|-----------------|
| Critical | App is down / data loss occurring        | 1 hour        | Same day        |
| High     | Major feature broken, no workaround      | 4 hours       | Within 1 day    |
| Medium   | Feature partially broken, workaround exists | 1 day      | Within 3 days   |
| Low      | Minor UI issue, cosmetic bug             | 3 days        | Next sprint     |

---

## 3. Support Contacts

| Role              | Contact Method     |
|-------------------|--------------------|
| Project Manager   | Team chat / email  |
| Lead Developer    | Team chat / GitHub |
| QA Engineer       | Team chat / email  |

---

## 4. Escalation Path

1. Issue reported → QA Engineer triages
2. If unresolved in response window → escalate to Lead Developer
3. If still unresolved → escalate to Project Manager
4. Critical issues → all team members notified immediately

---

## 5. Maintenance Windows

- Scheduled maintenance: Sundays 12:00 AM – 2:00 AM
- Users notified 24 hours in advance via email
- No maintenance during sprint demos or release windows
